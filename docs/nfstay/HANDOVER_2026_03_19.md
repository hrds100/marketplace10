## WHY CHANGES AREN'T VISIBLE ON nfstay.app

Two reasons:

1. **Branch not deployed**: Only `main` auto-deploys to production. The feature branch needs a PR + merge (Hugo approval required).

2. **nfstay.app is NOT a white-label site**: The hostname `nfstay.app` is detected as `type: 'main'` and renders `NfsMainSiteLayout` (traveler search/property pages). White-label layout only renders on **subdomains** like `brand.nfstay.app` or custom domains.

| Hostname | Mode | Layout |
|----------|------|--------|
| `nfstay.app` | `main` | NfsMainSiteLayout (traveler) |
| `brand.nfstay.app` | `subdomain` | NfsWhiteLabelLayout (Phase 1/2 work) |
| `bookings.example.com` | `custom` | NfsWhiteLabelLayout |
| `hub.nfstay.com` | `hub` | marketplace10 routes |
| `localhost:8080` | `dev` | marketplace10 routes |

### How to test white-label locally
1. Run `npm run dev`
2. Add to `/etc/hosts`: `127.0.0.1 demo.nfstay.app`
3. Visit `http://demo.nfstay.app:8080`
4. Need an operator in DB with `subdomain = 'demo'` and `landing_page_enabled = true`

## END-TO-END BOOKING FLOW (WORKING)

```
Traveler visits brand.nfstay.app
    ↓
Landing page (hero, about, FAQ, "Browse Properties" CTA)
    ↓
Search page (text + guest filter, property grid)
    ↓
Property detail (gallery, amenities, booking widget)
    ↓
Booking widget: select dates → check availability → see pricing → enter email
    ↓
"Book Now" → creates reservation (nfs_reservations) → creates Stripe checkout
    ↓
Redirect to Stripe hosted checkout
    ↓
Success → /payment/success?reservation_id=... (confirmation + booking link)
Cancel  → /payment/cancel?reservation_id=... (retry button)
    ↓
/booking/:id — view reservation details (from email link)
```

**Prerequisite**: Operator must have Stripe Connect connected (`/nfstay/settings`).
